## ⚡️ 1. Optimize Database (MongoDB) Queries

### ✅ 1.1. Use `.lean()` for all read queries

By default, Mongoose returns *full documents* with getters/setters/virtuals.
`.lean()` returns *plain JavaScript objects* — ~2–3x faster and lower memory use.

**Example:**

```ts
await this.playerService.find({ events: { $in: [team.event] } }).lean();
await this.matchService.find({ $or: [...] }).lean();
```

Add `.lean()` in all `.find()`, `.findOne()` calls that are read-only.

---

### ✅ 1.2. Ensure MongoDB indexes exist

Indexes drastically reduce query time.

Check these fields in your collections and **create indexes**:

| Collection      | Field(s) to Index        | Reason                         |
| --------------- | ------------------------ | ------------------------------ |
| `players`       | `events`, `teams`, `_id` | Filtering by `event` and `_id` |
| `groups`        | `_id`                    | Direct findOne                 |
| `matches`       | `teamA`, `teamB`         | `$or` query filters by these   |
| `rounds`        | `match`                  | You query rounds by match IDs  |
| `nets`          | `match`                  | Same reason                    |
| `playerstats`   | `player`, `net`, `match` | You query by player            |
| `playerRanking` | `team`, `match`          | You query by team & match      |

In Mongo shell:

```js
db.matches.createIndex({ teamA: 1 });
db.matches.createIndex({ teamB: 1 });
db.nets.createIndex({ match: 1 });
db.playerstats.createIndex({ player: 1 });
```

---

### ✅ 1.3. Replace multiple small queries with aggregation (joins)

You’re fetching a lot of related data (team → event → matches → rounds → nets).
MongoDB’s `$lookup` can combine them into a single round trip.

Example:

```ts
this.teamModel.aggregate([
  { $match: { _id: new ObjectId(teamId) } },
  {
    $lookup: {
      from: "matches",
      localField: "_id",
      foreignField: "teamA",
      as: "matches",
    },
  },
  // Add more lookups for players, events, etc.
]);
```

One aggregated query can replace several `Promise.all()` calls.

## 🚀 2. Optimize Redis Usage

### ✅ 2.1. Use Redis **mget / pipeline** instead of multiple `get` calls

You’re doing:

```ts
await Promise.all(redisKeys.map(key => this.redisService.get(key)));
```

That’s **N separate network calls**.

Use `mget` (or pipeline) for batch retrieval:

```ts
const redisResults = await this.redisService.mget(...redisKeys);
```

If using `ioredis`:

```ts
const pipeline = this.redisService.pipeline();
redisKeys.forEach(key => pipeline.get(key));
const redisResults = (await pipeline.exec()).map(r => r[1]);
```

This cuts Redis latency *dramatically* (especially for 50+ nets).

---

### ✅ 2.2. Cache team details & player stats

If these queries don’t change often, cache the **final computed response** in Redis:

```ts
const cacheKey = `teamDetails:${teamId}`;
const cached = await this.redisService.get(cacheKey);
if (cached) return JSON.parse(cached);

// ... heavy logic ...

await this.redisService.set(cacheKey, JSON.stringify(result), 'EX', 300);
```

This gives sub-millisecond retrieval for repeat queries.

## 🧮 3. Reduce the number of async DB calls

### ✅ 3.1. Avoid nested awaits

Right now, you’re calling:

```ts
await Promise.all(players.map(async player => {
  await this.playerService.updateOne(...);
  await this.playerStatsService.find(...);
}));
```

These nested async calls can overwhelm your database.
Use **bulk operations** instead of looping updates/finds.

Example: batch updates with MongoDB’s `bulkWrite()`:

```ts
const updates = players
  .filter(p => !p.username)
  .map(p => ({
    updateOne: {
      filter: { _id: p._id },
      update: { username: this.playerService.playerUsername(p.firstName) },
    },
  }));
if (updates.length) await this.playerService.bulkWrite(updates);
```

---

### ✅ 3.2. Preload everything you can

Instead of multiple `.find()` calls per collection, fetch **once** and use local filtering in memory.
Example: You’re fetching player stats per player; instead, fetch all and group by player in JS:

```ts
const allPlayerStats = await this.playerStatsService.find({ player: { $in: players.map(p => p._id) } }).lean();
const grouped = allPlayerStats.reduce((acc, s) => {
  (acc[s.player] ||= []).push(s);
  return acc;
}, {});
```

Now you don’t need per-player DB calls.

## 🧠 4. Parallelization & batching

You already use `Promise.all()`, which is good.
However, in places where you loop large arrays (players, matches), limit concurrency to avoid overloading MongoDB/Redis.

Use `p-limit`:

```ts
import pLimit from 'p-limit';
const limit = pLimit(10); // 10 concurrent players
await Promise.all(players.map(p => limit(async () => processPlayer(p))));
```

## 🧰 5. Architectural / Service-Level Improvements

### ✅ 5.1. Use DataLoader (for GraphQL)

If GraphQL resolvers call this query multiple times, use a **DataLoader** to batch identical DB calls per request.

```ts
const playerLoader = new DataLoader(async (ids) => {
  const players = await this.playerModel.find({ _id: { $in: ids } }).lean();
  return ids.map(id => players.find(p => p._id.equals(id)));
});
```

DataLoader avoids N+1 queries in GraphQL.

---

### ✅ 5.2. Use Redis as a read-through cache for Mongo

Wrap `findOne` and `find` calls with a caching decorator:

```ts
async findWithCache(collection: string, key: string, query: any) {
  const redisKey = `${collection}:${key}`;
  const cached = await this.redisService.get(redisKey);
  if (cached) return JSON.parse(cached);

  const data = await this[`${collection}Service`].find(query).lean();
  await this.redisService.set(redisKey, JSON.stringify(data), 'EX', 120);
  return data;
}
```

Then call `findWithCache('player', teamId, { event: team.event })`.

---

### ✅ 5.3. Precompute heavy data with background jobs

Some computations like `statsOfPlayer` can be precomputed periodically (e.g., every 1–5 minutes) and stored in Redis.

Use a worker (e.g., BullMQ):

* Worker fetches fresh data and caches in Redis.
* GraphQL just reads from Redis → instant response.

---

### ✅ 5.4. Use read replicas (if traffic is high)

If this query is called frequently, run MongoDB with a **replica set** and route heavy read queries to a **secondary replica**.

## 🧩 6. Smaller but useful tweaks

| Optimization            | Description                                                                                            |
| ----------------------- | ------------------------------------------------------------------------------------------------------ |
| **Pagination**          | If players, matches, or nets are large, don’t fetch all at once. Support pagination (`limit`, `skip`). |
| **Projection**          | Use `.find({}, 'field1 field2')` to fetch only needed fields.                                          |
| **Avoid `toObject()`**  | Use `.lean()` instead — it skips full doc conversion.                                                  |

## ⚙️ Example of Optimized Structure

Here’s a simplified structure applying several of the optimizations:

```ts
async getTeamDetails(teamId: string) {
  const cacheKey = `teamDetails:${teamId}`;
  const cached = await this.redisService.get(cacheKey);
  if (cached) return JSON.parse(cached);

  const team = await this.teamService.findById(teamId).lean();

  const [players, group, event, matches] = await Promise.all([
    this.playerService.find({ events: { $in: [team.event] } }).lean(),
    this.groupService.findOne({ _id: team.group }).lean(),
    this.eventService.findOne({ _id: team.event }).lean(),
    this.matchService.find({ $or: [{ teamA: teamId }, { teamB: teamId }] }).lean(),
  ]);

  // Fetch all related collections in parallel
  const matchIds = matches.map(m => m._id);
  const [rounds, nets, rankings] = await Promise.all([
    this.roundService.find({ match: { $in: matchIds } }).lean(),
    this.netService.find({ match: { $in: matchIds } }).lean(),
    this.playerRankingService.find({ team: teamId }).lean(),
  ]);

  // Preload all player stats in one go
  const playerIds = players.map(p => p._id);
  const [stats, redisStats] = await Promise.all([
    this.playerStatsService.find({ player: { $in: playerIds } }).lean(),
    this.redisService.mget(...playerIds.map(id => `playerstats:${id}`)),
  ]);

  const result = { team, players, group, event, matches, rounds, nets, rankings, stats };
  await this.redisService.set(cacheKey, JSON.stringify(result), 'EX', 300);
  return result;
}
```
